# AI와 페어프로그래밍: 잘 되는 것과 안 되는 것

> Claude Code를 실제 프로젝트에 적용하며 발견한 패턴과 한계

---

## 잘 되는 것

### 버그 디버깅

오류 메시지나 로그를 그대로 붙여넣으면 원인을 빠르게 파악합니다.

```
Type error: Cannot find name 'gzipSync'.
```

→ import 누락 즉시 감지 및 수정

### 반복적인 구현

페이지네이션, 폼 검증, API 연동처럼 패턴이 명확한 작업은 빠르게 완성합니다.

### 코드베이스 탐색

"이 버그의 원인이 어디 있을까?"라는 질문에 관련 파일을 찾아 분석합니다.

### 리팩토링

"이 컴포넌트를 더 간결하게 만들어줘"처럼 목적이 명확하면 잘 처리합니다.

---

## 안 되는 것 (주의 필요)

### 도메인 판단

비즈니스 맥락이 필요한 결정은 AI가 틀릴 수 있습니다.
"이 기능이 필요한가?"는 개발자가 판단해야 합니다.

### 장기적인 아키텍처 설계

현재 코드베이스 기준으로 작동하는 코드를 생성하지만,
6개월 후 확장성까지 고려한 설계는 별도로 검토가 필요합니다.

### 외부 서비스의 최신 스펙

학습 데이터 이후 변경된 API나 라이브러리 스펙은 틀릴 수 있습니다.
공식 문서와 비교 확인이 필수입니다.

---

## 효과적인 협업 패턴

### 계획 먼저, 구현 나중

```
복잡한 기능은 먼저 계획 검토 후 실행
→ 불필요한 구현을 막고 방향을 조율할 수 있음
```

### 목적 중심 요청

```
❌ "useState로 페이지 상태를 관리하고 slice로 나눠서..."
✅ "아티클 목록에 페이지네이션을 추가해줘"
```

구현 방법이 아닌 목적을 전달하면 더 적합한 방법을 선택합니다.

### 핵심 로직은 직접 검토

AI가 생성한 스트림 처리, 인증 흐름, 상태 관리 로직은
직접 읽고 이해해야 나중에 수정하거나 디버깅할 수 있습니다.

---

## 생산성 변화

| 작업 유형 | 체감 속도 |
|-----------|-----------|
| 반복적 UI 구현 | 3-5배 빠름 |
| 버그 디버깅 | 2-3배 빠름 |
| 새 기술 스택 학습 | 1.5-2배 빠름 |
| 아키텍처 설계 | 큰 차이 없음 |

---

## 결론

AI 페어프로그래밍의 핵심은 **AI가 잘하는 부분과 사람이 잘하는 부분을 구분하는 것**입니다.

반복적인 구현과 디버깅은 AI에게, 판단과 설계는 개발자가 담당할 때
가장 효과적인 협업이 됩니다.
